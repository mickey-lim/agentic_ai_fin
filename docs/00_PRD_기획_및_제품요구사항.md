## 1. 🚀 비즈니스 비전 및 목적 (Vision & Purpose)

### 1.1 문제 정의 (Problem Statement)
현재 기업 내 실무자들은 단일 거대언어모델(Cloud LLM)을 활용해 업무를 자동화하고 있으나, 다음과 같은 한계에 봉착했습니다.
1. **비용 효율성 한계**: 단순 분류나 요약에도 비싼 과금형 최신 모델(GPT-4o, Claude 3.5)을 호출하여 불필요한 IT 예산이 낭비되고 있습니다.
2. **보안의 딜레마**: 사내 중요 데이터나 민감 정보를 퍼블릭 클라우드 LLM 모델에 통째로 전송하는 데 대한 보안 컴플라이언스 우려가 큽니다.
3. **태스크 처리 지연(Sequential Block)**: 다단계 추론과 최신 정보 검색이 혼재된 복잡한 질문의 경우, 모델이 순차적으로 답을 생성하느라 심각한 병목(Latency)이 발생합니다.

### 1.2 제품 비전 (Product Vision)
> *"사용자의 단일 프롬프트를 지능적으로 분해하고, 보안과 속도가 보장된 로컬 온디바이스(Gemma 27B no_think) 에이전트 병렬 부대를 지휘하여 업무 시간을 절반 이하로 압축하는 지능형 오케스트레이션 시스템"*

<br>

## 2. 👥 타겟 사용자 및 페르소나 (Target Persona)

이 오케스트레이션 엔진이 최종적으로 완성되었을 때 이점을 누릴 타겟 고객군입니다.

- **[페르소나 A] 사내 재무 및 회계 담당자 (Finance / Operation Manager)**
  - **Needs**: 매달 쏟아지는 부서별 지출결의서 및 법인카드 내역(대외비 엑셀)을 분석하고, 특정 사내 결산 양식에 맞춰 새로운 엑셀 파일로 자동 병합/생성하고 싶다.
  - **Pain Point**: 기존 챗봇은 문서가 조금만 길어져도 중간 내용을 잊어버리며, 무엇보다 **재무 데이터는 철저한 사내 핵심 기밀이므로 클라우드 LLM(GPT, Claude)에 절대 업로드할 수 없다.**
- **[페르소나 B] 사내 보안 및 컴플라이언스 인프라 조직 (Security Admin)**
  - **Needs**: 사내 재무팀이 AI 어시스턴트를 도입하고 싶어하지만, 클라우드로 데이터가 유출되는 것을 차단해야 하므로 전면 로컬(On-device)에서 돌아가는 안전한 AI 분석 통제 환경이 필요하다.

<br>

## 3. 🎯 핵심 가치 제안 및 유저 스토리 (User Stories)

사용자 중심의 기능 에픽(Epic)과 기대 효과입니다.

### Epic 1. 지능형 테스크 분할을 통한 복잡한 엑셀(Excel) 가공 가속화 및 이미지 증빙 판독
- **사용자 스토리**: "재무 담당자로서 나는 '1분기 각 부서별 지출결의서 폴더 내 엑셀들과 종이 영수증 사진(JPG)들을 모두 읽고 요약해'라고 지시하면, 시스템이 VLM(Vision Language Model) 텍스트 추출과 엑셀 연산을 에이전트별로 쪼개어 병렬 수행한 완성본만 받아보고 싶다."
- **비즈니스 가치**: 사람이 건건이 엑셀과 영수증 사진을 열어 복사/붙여넣기 및 검증하던 '재무 대사 및 지출결의(Expense/Treasury 도메인 중심)' 업무 시간을 최소 50% 단축.

### Epic 2. 로컬 온디바이스 에이전트를 통한 완전한 재무 프라이버시(보안) 확보
- **사용자 스토리**: "재무 담당자 및 보안팀으로서 나는 사내 민감 계좌 및 지출 내역이 포함된 엑셀을 처리할 때, 초기 도입 시에는 Phase 1(Gemini API)로 더미 데이터 테스트를 마친 후, 실무 도입 시기엔 핵심 문서 열람과 추론이 망이 단절되거나 격리된 로컬 환경(Gemma 27B no_think) 내에서 안전하게 처리됨을 100% 보장받고 싶다."
- **비즈니스 가치**: 클라우드로의 정보 유출을 원천 차단함으로써, 금융/재무 데이터를 다루는 기업에서도 즉각적인 AI 도입의 컴플라이언스(NF-11~14) 심사 허들을 제거합니다. (Gemini -> Gemma 간의 부드러운 스위칭 활용)

### Epic 3. 적정 기술 배치로 인한 체감 운영 비용 절감
- **사용자 스토리**: "관리자로서 나는 단순 파싱 작업(Data Formatting)을 무거운 모델이 아니라 가벼운 모델(`Gemini Flash` 또는 `Gemma e2b`)로 자동 위임시켜, 사용자가 늘어나도 클라우드 API 청구 비용이 선형으로 폭증하지 않기를 바란다."

<br>

## 4. 📈 성공 지표 (Success Metrics)

PoC가 아닌 프로덕트(Product) 관점에서의 비즈니스 성공 판별 지표입니다. (엔지니어링 KPI인 NF-01~04와 별개 유지)

1. **태스크 수행 속도 향상** : 기존 인간 혹은 단일 챗봇 프로세스 대비 처리 시간이 얼마나 단축되었는가? (목표: 50% 단축)
2. **평균 처리 비용 (Cost per Query)** : 클라우드 API 전송량 감소로 인해 복합 질의 1건 당 원가가 하락했는가? (목표: 기존 대비 -30% 절감)
3. **태스크 완결성 및 유저 피드백 (User Satisfaction)** : 한 번의 프롬프트로 의도했던 모든 하위 태스크가 올바르게 재결합되어 최종 아웃풋으로 도출되는가? (통합 Pass Rate 85%)

## 5. 🔮 확장을 위한 미래 로드맵 (Future Roadmap)
금번 PoC(v0.2)에서는 터미널/API 형태로 검증되지만, 도입 확정 시 다음 단계로 이행합니다.

- **Phase 1 (현재, v0.2)**: 분해-위임-취합 파이프라인 엔진 완성 (API 관점 검증)
- **Phase 2 (차수 확장)**: 사내 DB 조회 (SQL 에이전트 결합), LangGraph Human-in-the-loop (승인 절차 엔진) 붙임.
- **Phase 3 (Front-end 릴리스)**: 투-트랙 UI (오른쪽엔 채팅창, 왼쪽엔 에이전트들이 분해/수행하는 과정을 실시간 트레이싱으로 보여주는 Glassmorphism 기반의 미션 컨트롤 대시보드 UI 오픈).
