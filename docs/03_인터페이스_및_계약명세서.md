## 1. 에이전트 인터페이스 계약 (Interface Contracts)

오케스트레이터(`gemini-2.5-flash`)와 서브에이전트 계층 사이의 데이터 교환은 **명시적이고 강제된 JSON 스키마 Envelope** 에 절대적으로 의존합니다.

### 1.1 Workflow Envelope (Orchestrator Output)
오케스트레이터가 서브태스크 배열(Task List)을 생성하여 위임할 때 사용하는 최상위 스키마입니다.
```json
{
  "workflow_id": "uuid",
  "process_family": "vat|withholding|payroll|social_insurance|expense|treasury|corporate_tax|grant|unknown",
  "submission_channel": "hometax|nps_edi|4insure|tax_agent|manual",
  "legal_owner": "manager|team_leader|ceo|tax_agent",
  "approval_mode": "auto_bypass|human_review|external_handoff",
  "evidence_bundle_required": true,
  "sensitivity_level": "high|low|human",
  "tasks": [
    // TaskEnvelope...
  ]
}
```

### 1.2 Task Envelope (Orchestrator ➔ Sub-agent)
위 `Workflow Envelope`의 `tasks` 내역이 각각의 워커로 할당될 때 구체화되는 개별 계약입니다.
```json
{
  "task_id": "uuid",
  "task_type": "collect|normalize|draft|package|compute",
  "ai_type": "AI_주도|AI_보조|사람_전담",
  "target_tier": "high|low|human",
  "depends_on": ["parent_task_id"],
  "goal": "string",
  "inputs": {
    "key": "value (task specific raw data)"
  },
  "tools_allowed": ["excel_parser", "execute_python", "file_io"]
}
```

### 1.3 Result Envelope (Sub-agent ➔ Orchestrator)
```json
{
  "task_id": "uuid",
  "status": "success|failed|partial",
  "output": {
    "key": "computed result value"
  },
  "evidence": ["source file path", "formula used"],
  "cost": {
    "tokens_in": 0,
    "tokens_out": 0
  },
  "latency_ms": 0,
  "error": {
    "code": "e.g., TOOL_RUNTIME_ERROR",
    "message": ""
  }
}
```

## 2. 에러 대응 및 실패 정책 (Failure & Retry Policies)

서브에이전트 가동 중 발생 가능한 에러 시나리오. **모든 장애 제어는 `Loop Guard` 시스템에 의해 최대 2회 재시도로 엄격하게 제한됩니다.**

### 2.1 Format Error 및 스키마 위반
LLM(Planner) 혹은 어댑터가 지정된 `Result Envelope` 구조 대신 빈 문자열이나 잘못된 JSON 형식을 뱉는 경우입니다.
- **정책 (Policy)**: JSON Output 파싱 단계에서 에러 포착 시 **1회 내부 재시도** 수행.
- **수행 액션**: 오케스트레이터가 `"You MUST respond strictly in valid JSON format ONLY. Do NOT wrap with markdown."` 프롬프트를 주입하여 자가 교정.
- **예방 (Prevention)**: Planner 출력에 Pydantic의 `Structured Output` 검증을 강제하여 사전 차단.

### 2.2 Timeout Error
워커가 주어진 `timeout_sec` 이내에 결과값을 반환하지 못하는 경우입니다.
- **정책 (Policy)**: 1회 재시도. 실패 시 `Partial Success`로 처리하거나 실패로 리포팅.
- **수행 액션**: 어댑터 로직 타임아웃 오류 시 스냅샷 보호를 위해 즉시 실패(Failed)로 1회 리포팅 후 부분 재시도 허용 검토.

### 2.3 Rate Limit Error (HTTP 429)
`Gemini API` 등 외부 API의 RPM 호출 제한에 걸리는 경우입니다.
- **정책 (Policy)**: Exponential Backoff 적용.
- **수행 액션**: 5초 대기 후 1회 재시도, 실패 시 15초 대기 후 1회 추가 재시도.

### 2.4 Tool Runtime Error & File Not Found
샌드박스에서 파이썬 코드가 실패하거나, 지정된 대상 엑셀 파일이 존재하지 않는 경우입니다.
- **정책 (Policy)**: File Not Found는 즉시 실패 처리. 런타임 에러 발생 시 Python Traceback을 로깅하고 시스템 격리 후 실패(Failed) 상태로 반환.
- **수행 액션**: 이중 루프 오동작(무한 루프) 방지를 위해 파이프라인 중단 및 결함 원인(Traceback)을 Registry에 기록하여 알림(Alert) 발행.

### 2.5 Partial Success (부분 성공)
일부 서브태스크는 성공했으나 전체 프로세스 중 일부가 실패한 경우입니다.
- **정책 (Policy)**: 성공한 데이터를 기반으로 워크플로우를 진행하되, 관리자 리뷰(Human-in-the-loop) 큐로 이관.
- **수행 액션**: `status: "partial"` 처리 후 수집된 부분 데이터를 리포트에 별도 표기.

## 3. 프론트엔드-백엔드 API 계약 및 방어 경계 (Phase 5 Hardening)

운영 환경(Production)의 안정성을 보장하기 위해 프론트엔드와 백엔드 간의 통신 계약에 강력한 방어선이 추가되었습니다.

### 3.1 워크플로우 통신 계약 (2-Stage Upload & Multi-file)
- **제 1단계 (단일 업로드 반복)**: 프론트엔드는 사용자가 첨부한 복수의 파일(Excel, CSV, PDF, JPG, PNG)을 각각 독립적으로 `/workflows/upload` 엔드포인트에 순차 전송하여 개별 `file_id` 들을 수집합니다.
- **제 2단계 (다중 파일 결합 시작)**: 모아진 `file_id` 배열을 `/workflows/start`의 `source_file_ids: List[str]` 페이로드로 한 번에 전달하여 단일 쓰레드로 묶습니다.
- **프롬프트 안전 제한**: `input_request`는 메모리 고갈을 막기 위해 **최대 15,000자**로 제한(`max_length=15000`)됩니다.

### 3.2 VLM(이미지 판독) 지원 도메인 규격 (Scope limits)
- **허용 범위 한정**: Gemini VLM을 이용한 비정형 이미지 판독은 "이미지/스캔 문서 전반"을 커버하지 않으며, 현재는 **영수증, 지출결의서 형태의 Expense/Treasury (비용/자금) 도메인에 한하여 타겟팅 최적화**되어 있습니다. 그 외 문서의 판독은 제한적 지원 및 Fallback 대상입니다.

### 3.3 Evidence Preview (증빙 데이터) API 제한
- **계약**: UI 패널에서 엑셀 스냅샷 프리뷰를 렌더링하기 위해 `/workflows/{thread_id}/evidence` 엔드포인트 호출 시, 원본 수백만 행의 파일을 통째로 JSON으로 직렬화하지 않고 **상위 100건(head=100)**의 행렬 데이터만 전달합니다.
- **방어 기제**: 프론트엔드 ব্র라우저의 DOM 렌더링 블로킹 방어 및 API 서버의 메모리 OOM 예방. 원본 데이터는 UI가 아닌 직접 `.zip` 다운로드를 통해 확보하는 것을 원칙으로 합니다.
