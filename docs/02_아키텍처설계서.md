# Agentic AI Orchestration PoC (v1.0-rc.1) - 02. 아키텍처 설계서

**프로젝트명**: Agentic AI Orchestration PoC  
**소속**: Google Antigravity  
**문서 버전**: v1.0-rc.1 (Agentic Planner + Celery Async Pipeline + Next.js UI Hardened)

## 1. 시스템 컴포넌트 아키텍처 (System Component Architecture)

초기의 단일 스레드 구조를 넘어서서, 안정적인 엔터프라이즈 워크플로우 처리를 위해 **FastAPI 제어 평면**, **Redis 브로커**, **Celery 분산 워커 계층**, 그리고 **Next.js 기반 HITL 관제 대시보드**로 분리된 완벽한 하이브리드 비동기 아키텍처를 구현하고 있습니다.

### 1.1 오케스트레이터 및 제어 평면 (FastAPI & Planner)
- **제어 평면 (FastAPI)**: 사용자 업로드 및 Batch 운영(`delete`, `restore`, `purge`) 요청을 수신하고, 상태 모델(Registry DB)을 갱신한 후 즉시 응답 반환(Non-blocking).
- **오케스트레이터 모델**: **`gemini-2.5-flash`** (빠르고 효율적인 추론)
- **담당 역할**: 자연어 또는 구조화된 데이터를 수집하여 종속성이 보장된 Task 단위로 분석(MECE)하고 라우팅 계획을 수립합니다.

### 1.2 비동기 태스크 큐 및 브로커 (Celery & Redis)
- **메시지 브로커**: `Redis` (인메모리 기반 고속 큐 시스템 지원)
- **비동기 큐 런타임**: `Celery`
- **담당 역할**: FastAPI가 던진 워크플로우 구동 티켓을 백그라운드 큐에 넣어 안정적으로 워커 노드에 분배합니다. 장애 발생 시 재처리(Retry)와 OOM 방지(`max-tasks-per-child=50`)를 관리합니다.

### 1.3 서브에이전트 계층 (Worker Node & Adapters)
- **런타임**: `Python 3.11+` 기반의 결정적(Deterministic) 함수 실행
- **목적**: Planner가 작성한 계획을 순차적으로 이어받아 LLM 환각(Hallucination) 없이 가장 정확하게 데이터를 파싱 및 가공합니다.
- **방어 메커니즘**: 파일시스템 기반의 Immutable Snapshotting으로 각 단계마다 독립적인 파일 I/O를 가져가 불변성을 유지합니다.

### 1.4 관제 대시보드 (Next.js App Router)
- **프론트엔드 관제 뷰**: `React (Next.js)` 기반 **Workflow Console**. 
- **담당 역할**: 워크플로우 파이프라인의 실시간 추이(Polling) 추적, 도출된 Evidence를 대조하며 휴먼 인 더 루프(HITL) 체제에서 최종 승인(Approve/Reject) 및 중단(Interrupted) 해결을 담당합니다.

## 2. 프레임워크 결정 및 인프라 (Framework & Infrastructure)

- **그래프 오케스트레이션**: **`LangGraph`**. 노드 기반의 State 그래프로 가시성이 높고 분기 제어가 강력함.
- **트레이싱 (옵션)**: **`LangSmith`** (API 토큰 소모량 및 에러율 모니터링).
- **구조적 로깅 (Observability)**: `APP_ENV=prod` 시 JSON 포맷 출력, `APP_ENV=local` 시 Pretty CLI 로깅을 사용하여 Kibana 등에서 기계적 파싱이 용이한 Observability를 제공.

<br>

## 3. [Future Archive] Local 핫스왑 트랜지션 로드맵 (Security)

기밀 파일에 직접 접근하는 파이프라인(Phase 2)으로 나아갈 경우의 구조적 대비안입니다.

- **R-01 (클라우드 유출 방어)**: 
  > 현재 설계된 워크플로우의 노드 결합도를 낮춰 `ChatGoogleGenerativeAI` 래퍼 대신 `ChatOllama`나 MLX 기반의 로컬 래퍼로 스위칭 할 수 있도록 설계되어 있습니다.
- **R-02 (로컬 추론 시 포맷/파싱 실패)**:
  > 오프라인 환경 구동 시, 모델 타겟을 **`Gemma 27B (no_think / Instruct)`** 등 툴 콜링 지시어를 철저히 지키는 클린 모델로 고정하여 핫스왑 런타임을 무결하게 유지합니다.

## 4. 데이터 영속성 및 방어 아키텍처 (Data Integrity & Mitigation)

프로덕션 운영 및 재배포 시의 시스템 붕괴 및 병목을 막기 위해 큐 기반 시스템 레벨 방어 로직을 통합했습니다.

### 4.1 상태 보존 전략 (Volume & Persistence)
- **Checkpointer DB & Registry DB**: LangGraph 상태(`agentic_state.db`)와 FastAPI 큐 상태(`agentic_registry.db`)가 존재합니다.
- **다중 파일 병합 스키마 (Canonical Relation)**: 
  - 워크플로 1개가 N개의 파일을 가질 수 있도록 `workflow_source_files` M:N 조인 테이블을 **진실의 원천(Canonical Relation)**으로 사용합니다.
  - 기존 `workflow_registry`의 `source_file_id` 컬럼은 과거 데이터 역호환(Backward Compatibility) 용도로만 남겨 가비지 컬렉터(GC)에서 `UNION` 탐색 시 보조적으로 활용합니다.

### 4.2 SQLite 동시성 로킹 해제 (Concurrency Hardening)
- **WAL & PRAGMA**: 다중 Celery Worker 와 FastAPI가 동시에 SQLite 파일을 엑세스 할 때 발생하는 `database is locked` 문제를 발본색원하기 위해 아래 설정을 강제 주입했습니다.
  - `PRAGMA journal_mode=WAL;` (Write-Ahead Logging 체제로 읽기/쓰기 동시성 보장)
  - `PRAGMA synchronous=NORMAL;`
  - `PRAGMA busy_timeout=5000;`

### 4.3 불변 스냅샷 방어 (Immutable Snapshotting)
- **비연속적 데이터 오염 방지**: `worker.py` 내부 태스크는 외부 데이터를 재수집하지 않습니다. 한 번 가공되어 `tests/fixtures/` 등에 생성된 `.xlsx` Evidence 스냅샷을 물고 다음 Graph 노드로 이동하므로 HITL 대기 중 데이터의 결산 일관성이 강제 유지됩니다.
