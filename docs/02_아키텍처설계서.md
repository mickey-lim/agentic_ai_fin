# Agentic AI Orchestration PoC (v1.0-rc.1) - 02. 아키텍처 설계서

**프로젝트명**: Agentic AI Orchestration PoC  
**소속**: Google Antigravity  
**문서 버전**: v1.0-rc.1 (Agentic Planner + Deterministic Pipeline + Next.js UI Hardened)

---

## 1. 시스템 컴포넌트 아키텍처 (System Component Architecture)

비율 제한(Rate Limit) 병목과 데이터 보안 문제를 완벽히 해결하기 위해, 현재 시스템은 **Planner (LLM)**와 **결정적 Worker (Python Adapter 파이프라인)**, 그리고 **Next.js UI & FastAPI**로 이루어진 단일/하이브리드 오케스트레이션 아키텍처를 구현하고 있습니다.

### 1.1 오케스트레이터 계층 (Planner Node)
- **코어 모델**: **`gemini-2.5-flash`** (빠르고 효율적인 추론)
- **담당 역할**: 사용자의 자연어 명령을 수집하여 `collect`, `normalize`, `draft`, `package` 순서의 종속성이 보장된 Task 단위로 분석(MECE)하고, `WorkflowEnvelope` 형태의 JSON 궤적으로 라우팅 계획을 수립합니다.

### 1.2 서브에이전트 계층 (Worker Node & Adapters)
- **런타임**: `Python 3.11+` 기반의 결정적(Deterministic) 함수 실행
- **목적**: Planner가 작성한 계획을 순차적으로 이어받아 LLM 환각(Hallucination) 없이 가장 정확하게 데이터를 추출(collect)하고 포맷팅(draft)합니다.
- **방어 메커니즘**: 파일시스템 기반의 Immutable Snapshotting 체계로 이전 노드의 산출물을 읽어들입니다.

### 1.3 어드민 대시보드 (Next.js & FastAPI)
- **프론트엔드 제어 평면**: `React (Next.js App Router)` 기반의 **Workflow Console**. 
- **담당 역할**: 생성된 워크플로우를 모니터링하고, 도출된 Evidence를 프리뷰하며 휴먼 인 더 루프(HITL) 체제에서 최종 승인(Approve/Reject)을 담당하는 관리자 인터페이스입니다.
- **State Management**: LangGraph 내부 Checkpointer(`agentic_state.db`) 및 FastAPI UI Registry(`agentic_registry.db`)의 볼륨 마운트 연계를 통한 완벽한 상태 영속성.

## 2. 프레임워크 결정 (Framework)

- **프레임워크**: **`LangGraph`** (우선 타겟). 노드 기반의 State 그래프로 가시성이 높으며, Gemini API 및 Local 언어 모델(Ollama/MLX) 교체가 가장 유연하게 구현되는 업계 표준.
- **트레이싱**: **`LangSmith`** (API 토큰 소모량, 시간, 에러율 모니터링).

<br>

## 3. [Future Archive] Local 핫스왑 트랜지션 로드맵 (Security)

기밀 파일에 직접 접근하는 파이프라인의 완성 시점(Phase 2)의 구조적 대비안입니다.

- **R-01 (클라우드 유출 방어)**: 
  > 현재 설계된 LangGraph 워크플로우의 노드 결합도를 낮춰 `ChatGoogleGenerativeAI` 래퍼 대신 `ChatOllama` 혹은 MLX 커스텀 래퍼로 갈아끼우도록 설계합니다. 
- **R-02 (로컬 추론 시 포맷/파싱 실패)**:
  > 오프라인 환경 구동 시, 로컬 모델이 추론 과정을 뱉어내는 `<think>` 구문들 때문에 LangGraph의 JSON Output Parser 파이프가 붕괴되는 현상을 사전에 억제합니다. 즉, 로컬 모델 타겟을 무조건 **`Gemma 27B (no_think / Instruct)`** 형태의 클린 아웃풋 버전으로 강제 고정하여 핫스왑(Hot-Swap) 런타임을 무결하게 유지합니다.

## 4. 데이터 영속성 및 방어 아키텍처 (Data Integrity & Mitigation - Phase 5)

프로덕션 운영 및 재배포 시의 시스템 붕괴를 막기 위해 시스템 레벨의 방어 로직을 통합합니다.

### 4.1 상태 보존 전략 (Volume & Persistence)
- **Checkpointer DB & Registry DB 분리 매핑**: LangGraph의 상태(Checkpointer)와 커스텀 프론트엔드 대시보드 상태(Registry)는 분리되나, 모두 호스트의 영속적 `/data` 볼륨(`agentic_state.db`, `agentic_registry.db`)으로 마운트됩니다. 컨테이너가 교체되거나 재생성되어도 Approval Queue와 워크플로우 메타데이터가 완벽히 보존됩니다.

### 4.2 불변 스냅샷 파이프라인 (Immutable Snapshotting)
- **비연속적 데이터 오염 방지**: `worker.py` 내부 태스크(`normalize`, `draft`, `package`)는 외부 데이터를 재수집하지 않습니다. 각 워커 노드는 `t["depends_on"]`에 기록된 직전 단계의 태스크 ID를 역추적하여, 물리 디스크에 안착된 엑셀 파일(`/evidence/{prev_task_id}_{workflow_id}.xlsx`)만을 읽어들입니다. 이로써 휴먼 인 더 루프(HITL) 대기 중 소스 데이터가 변경되더라도 결산 일관성이 훼손되지 않습니다.

### 4.3 블로킹 및 과부하 방지 (Dos / OOM Protection)
- **비동기 블로킹 처리 (Asyncio Threading)**: FastAPI의 메인 루프를 블로킹할 수 있는 무거운 `pandas.read_excel()` 등의 I/O 연산은 `await asyncio.to_thread`를 사용하여 별도 스레드풀로 오프로딩시켰습니다.
- **Payload Truncation**: `/evidence` 엔드포인트는 프론트엔드 DoS 방어를 위해 `df.head(100)`으로 데이터프레임을 잘라서 전송합니다. 전체 파일은 `.zip` 아카이빙을 통해서만 내려받게 하여 VRAM 및 브라우저 메모리 고갈을 방어합니다.
